# Key Concepts

## The Two Dependency Files

uv uses two files to manage dependencies. Think of it as a **recipe vs a grocery receipt**:

### `pyproject.toml` вҖ” what you want

- You write and maintain this file
- Declares your **direct** dependencies with version ranges
- Also holds project metadata (name, Python version, test config, etc.)

```toml
[project]
name = "my-api"
requires-python = ">=3.13"
dependencies = [
    "fastapi>=0.123.0",
    "sqlalchemy>=2.0",
]
```

### `uv.lock` вҖ” what you actually get

- Auto-generated by `uv lock` or `uv sync` вҖ” **never edit manually**
- Pins **every** package (including transitive dependencies) to exact versions
- Ensures reproducible installs across machines

For example, you declared 2 packages in `pyproject.toml`, but `uv.lock` might resolve to 20+ packages because fastapi depends on starlette, which depends on anyio, and so on.

### Summary

| | `pyproject.toml` | `uv.lock` |
|---|---|---|
| **Who writes it** | You | uv (auto-generated) |
| **Purpose** | Declare your intent | Record the exact resolution |
| **Versions** | Ranges (`>=1.0`) | Exact pins (`==1.2.3`) |
| **Includes transitive deps** | No | Yes |
| **Commit to Git?** | Yes | Yes |

:::tip Always commit both files
`pyproject.toml` without `uv.lock` means teammates might get different versions. `uv.lock` without `pyproject.toml` means nobody knows which dependencies are direct vs transitive. Both files together give you intent + reproducibility.
:::

## Where does `requirements.txt` fit?

`requirements.txt` is the legacy way of doing both jobs вҖ” poorly. It's a flat list with no distinction between what you want and what you got:

```txt
# requirements.txt вҖ” is "starlette" my dependency or fastapi's?
fastapi==0.123.0
starlette==0.41.2
anyio==4.3.0
pydantic==2.5.0
```

With uv, you don't need `requirements.txt`. If you're deploying to a system that only understands `requirements.txt` (e.g., some cloud platforms), you can generate one:

```bash
uv pip compile pyproject.toml -o requirements.txt
```

## Dependency Resolution

This is where uv's speed shines вҖ” it resolves dependencies 10вҖ“100x faster than pip because the resolver is written in Rust.

## Virtual Environments

A virtual environment is an **isolated Python installation** for your project. It prevents dependency conflicts between projects вҖ” e.g., Project A needs `fastapi==0.100.0` but Project B needs `fastapi==0.123.0`.

### How uv handles venvs

uv creates a `.venv` directory in your project root. Most uv commands handle this automatically:

| Command | Creates venv if missing? | Uses the venv? |
|---|---|---|
| `uv sync` | Yes | Yes |
| `uv run <cmd>` | Yes | Yes |
| `uv add <pkg>` | Yes | Yes |
| `uv venv` | Yes (explicit) | вҖ” |

:::note
You only need to activate the venv (`source .venv/bin/activate`) if you want to use `python` or `pip` directly in your shell. When using `uv run`, activation is not needed вҖ” uv injects the venv automatically.
:::

## `uv pip install` vs `uv sync` вҖ” When to Use Which

These are two different workflows and should not be mixed:

### Project workflow (recommended)

Uses `pyproject.toml` + `uv.lock`. This is what you should use for any real project.

```bash
uv add fastapi        # add dependency to pyproject.toml + install
uv sync               # install everything from lockfile
uv run python app.py  # run using venv
```

### pip-compatibility workflow

A drop-in replacement for pip commands. Useful for quick scripts, CI environments that expect pip, or legacy projects not yet migrated to `pyproject.toml`.

```bash
uv pip install fastapi                # install a package (no lockfile)
uv pip install -r requirements.txt    # install from requirements.txt
uv pip freeze                         # list installed packages
```

:::caution Don't mix the two
If your project uses `pyproject.toml` and `uv.lock`, stick to `uv add` / `uv sync`. Using `uv pip install` bypasses the lockfile and can cause your venv to drift out of sync with `uv.lock`.
:::
