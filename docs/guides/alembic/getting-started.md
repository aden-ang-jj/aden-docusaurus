# Getting Started with Alembic

## Prerequisites

```bash
# Using uv (recommended)
uv add alembic sqlalchemy

# Or using pip
pip install alembic sqlalchemy
```

## Project Setup

```bash
# Initialize Alembic in your project
alembic init alembic
```

This creates:

```
your-project/
├── alembic.ini              # Config file (database URL, logging)
├── alembic/
│   ├── env.py               # Connects Alembic to your models
│   ├── script.py.mako       # Template for new migration files
│   └── versions/            # Migration scripts live here
```

:::caution Folder name collision
The default `alembic init alembic` creates a folder called `alembic/`. If your `PYTHONPATH` includes the parent directory, this folder can **shadow the installed `alembic` package**, causing `ModuleNotFoundError: No module named 'alembic.config'`. This is a common issue in Dockerized setups where `PYTHONPATH` is configured broadly.

To avoid this, name the folder `migrations` instead:

```bash
alembic init migrations
```

Then update `alembic.ini` to point to it:

```ini
script_location = %(here)s/migrations
```
:::

## Configuration

### Option 1: Hardcode the URL (simple, for local dev)

In `alembic.ini`, set your database URL:

```ini
sqlalchemy.url = postgresql://user:pass@localhost/mydb
```

### Option 2: Read from environment variables (recommended)

Hardcoding credentials in `alembic.ini` is fine for learning, but in real projects you should read the database URL from an environment variable. This keeps secrets out of version control and works across environments (dev, staging, production).

Leave `alembic.ini` with a fallback:

```ini
# Real URL comes from DATABASE_URL env var (see env.py)
sqlalchemy.url = postgresql://user:pass@localhost/mydb
```

Then in `env.py`, override it from the environment:

```python
import os

config = context.config

# Override with env var if available
database_url = os.environ.get("DATABASE_URL")
if database_url:
    config.set_main_option("sqlalchemy.url", database_url)
```

This way, `DATABASE_URL` from your environment (or `.env` file) takes priority, and the `alembic.ini` value is only a local fallback.

### Point Alembic to your models

In `env.py`, import your models' metadata so Alembic can detect changes:

```python
# For SQLAlchemy (declarative base)
from myapp.models import Base
target_metadata = Base.metadata

# For SQLModel
from sqlmodel import SQLModel
# Import all your models so they register with SQLModel.metadata
from myapp.infrastructure.users import UserModel  # noqa: F401
from myapp.infrastructure.posts import PostModel  # noqa: F401
target_metadata = SQLModel.metadata
```

:::note Why the "unused" imports?
SQLModel (and SQLAlchemy) only knows about a table after the model class has been **imported and executed**. If you skip the import, Alembic won't see the table and `--autogenerate` will produce empty migrations. The `# noqa: F401` comments tell linters to ignore the "unused import" warning.
:::

### SQLModel: Fix the migration template

If you use SQLModel, autogenerated migrations will reference `sqlmodel.sql.sqltypes.AutoString()` but the generated file won't include `import sqlmodel`, causing a `NameError` at runtime.

Fix this by editing `script.py.mako` (the template for new migration files):

```python title="alembic/script.py.mako"
from alembic import op
import sqlalchemy as sa
import sqlmodel  # ← add this line
${imports if imports else ""}
```

This ensures every future migration file automatically includes the `sqlmodel` import.

## The Typical Workflow

```
1. Update model code        →  edit your Python model file
2. Generate migration       →  alembic revision --autogenerate -m "message"
3. Review the script        →  check the generated file in alembic/versions/
4. Apply the migration      →  alembic upgrade head
```

:::note
Running `alembic` by itself only prints help text. You always need a subcommand like `upgrade`, `downgrade`, `revision`, etc.
:::

## Core Commands

| Command | Description |
|---|---|
| `alembic init alembic` | Scaffold the Alembic directory structure |
| `alembic revision --autogenerate -m "message"` | Auto-generate a migration by comparing your models to the DB |
| `alembic revision -m "message"` | Create an empty migration to write manually |
| `alembic upgrade head` | Apply **all** pending migrations (most common) |
| `alembic upgrade +1` | Apply only the **next** migration |
| `alembic upgrade a1b2c3` | Apply up to a **specific** revision ID |
| `alembic downgrade -1` | Revert the last migration |
| `alembic downgrade base` | Revert **all** migrations |
| `alembic stamp head` | Mark the DB as up-to-date **without running** migrations |
| `alembic current` | Show which migration the DB is currently at |
| `alembic history` | Show the full migration history |

## What does "head" mean?

`head` refers to the latest migration in the revision chain — borrowed from Git terminology:

```
(base) → a1b2c3 → d4e5f6 → g7h8i9 (head)
```

If your DB is at `a1b2c3` and you run `alembic upgrade head`, Alembic walks the chain and applies `d4e5f6` then `g7h8i9` in order. In practice, `alembic upgrade head` is what you'll use 99% of the time. The `+1` and specific revision options are useful for debugging — e.g. applying migrations one at a time to find which one breaks.

## Adopting Alembic on an Existing Database

If your database already has tables (e.g., created by `SQLModel.metadata.create_all()` or `Base.metadata.create_all()`), you can't just run `alembic upgrade head` — the migration would try to create tables that already exist.

The solution is to generate the initial migration, then **stamp** it as already applied:

```bash
# 1. Generate a migration that captures all existing tables
alembic revision --autogenerate -m "initial schema"

# 2. Mark it as applied WITHOUT running it (tables already exist)
alembic stamp head

# 3. Verify
alembic current
# Should show the revision ID of your initial migration
```

From this point on, any new model changes will produce incremental migrations that you apply normally with `alembic upgrade head`.

:::tip What does `stamp` do?
`alembic stamp head` writes the revision ID into the `alembic_version` table without executing any migration code. It's saying: "the database is already at this state, just record it."
:::

## Running Alembic in Docker

In Dockerized projects, you typically run Alembic commands inside the container that has your app's dependencies and network access to the database:

```bash
# Run inside an existing container
docker compose exec api alembic upgrade head
docker compose exec api alembic revision --autogenerate -m "add email to users"

# Or as a one-off container
docker compose run --rm api alembic upgrade head
```

This is the production-standard approach because:
- The container has the **same environment** (Python version, dependencies, network) as your app
- The database hostname (e.g., `postgres`) resolves correctly within the Docker network
- No need to install Python or Alembic on your host machine

In CI/CD pipelines, migrations are typically run as a one-off container before the app starts — similar to how Ory Kratos or Django run their migrations at deployment time.

## Exercise: Write a Migration

Given this model:

```python
class BlogPost(Base):
    __tablename__ = "blog_posts"
    id = Column(Integer, primary_key=True)
    title = Column(String(200), nullable=False)
    content = Column(Text)
    created_at = Column(DateTime, server_default=func.now())
```

Suppose you need to add a `status` column (draft/published) to `BlogPost`. Write the migration:

```python
def upgrade():
    op.add_column('blog_posts', sa.Column('status', sa.String(), nullable=False, server_default='draft'))

def downgrade():
    op.drop_column('blog_posts', 'status')
```
